# SRA-003-16: 架构优化 — 并发安全 + 路由统一

## 用户故事

> As a **SRA 守护进程维护者**,
> I want **多线程场景下状态一致、请求路由统一**,
> So that **在高并发下不丢失数据、新增端点只需修改一处**.

---

## 验收标准

### 🛡️ AC-1: `_update_status()` 加锁保护
- [ ] 条件: `_update_status()` 方法写 status 文件时使用 `self._lock`（复用现有锁）
- [ ] 验证: 两线程同时调用 `_update_status()` 不出现交叉写入
- [ ] 预期: status 文件内容始终完整有效（非 `\n` 截断）

### 🛡️ AC-2: `memory.py` 文件锁
- [ ] 条件: `memory.py` 的 `save()` 方法增加 `fcntl.flock` 文件锁
- [ ] 验证: 两个进程同时写 memory 文件时，顺序执行而非交叉写入
- [ ] 预期: memory 文件内容始终合法 JSON

### 🛡️ AC-3: `_last_refresh` 原子化
- [ ] 条件: `_last_refresh` 读写统一通过 `threading.Lock` 保护
- [ ] 验证: 高并发场景下多线程读 `_last_refresh` 返回一致值
- [ ] 预期: 无反重入 bug（前一个 refresh 未完成时后一个跳过）

### 🛡️ AC-4: `self._stats` 统一锁保护
- [ ] 条件: 所有 `self._stats` 的更新和读取通过 `self._lock`（或专用 `_stats_lock`）保护
- [ ] 验证: 并发请求下 stats 计数不丢失
- [ ] 预期: 100 次并发请求后总计数 = 100

### 🛡️ AC-5: Socket + HTTP 路由统一
- [ ] 条件: 提取 `ROUTER = {"recommend": ..., "record": ..., "refresh": ...}` 统一路由表
- [ ] 验证: Socket `/action` 和 HTTP `POST /{action}` 路由同一个函数
- [ ] 预期: 新增端点只需在 `ROUTER` 注册一次，无需修改两处分发逻辑

### 🛡️ AC-6: Socket 和 HTTP 端端到端一致性
- [ ] 条件: 所有现有端点通过 Socket 和 HTTP 返回相同结构和状态码
- [ ] 验证: 分别通过 Socket 和 HTTP 调用所有端点，比较响应结构
- [ ] 预期: Socket 响应字段 ⊆ HTTP 响应字段, HTTP 状态码语义正确

### 🛡️ AC-7: 回归测试通过
- [ ] 条件: 所有修改不破坏已有功能
- [ ] 验证: `python -m pytest tests/ -q --tb=short -o "addopts="`
- [ ] 预期: 304+ passed（新测试不计入基线，但旧测试全部通过）

## 技术要求

- 锁策略：`threading.Lock` 用于进程内并发，`fcntl.flock` 用于跨进程文件同步
- 路由表：`ROUTER` 为模块级 `dict`，key 为 action 名称（字符串），value 为可调用（函数引用）
- 路由表支持函数和 `(handler, methods)` 元组（HTTP 兼容）
- Socket handler 从 `ROUTER` 中查找 action，不再自行 switch
- HTTP handler 从 `ROUTER` 中查找路径对应的 action，统一入口
- 所有变更保持向下兼容（不改变现有 API 签名）
